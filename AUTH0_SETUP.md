# Auth0 Tenant Setup

Manual configuration reference for the bowling-tourney-service Auth0 tenant.
Keep this document updated whenever a tenant change is made.
When Terraform management is introduced, the sections below map directly to
`auth0_action`, `auth0_client`, and `auth0_resource_server` resources.

---

## Architecture decisions

| Decision | Rationale |
|---|---|
| No Organizations used | Free tier caps at 5 orgs — unusable for tournament scale |
| Auth0 = identity only | Team membership lives in the DB; Auth0 only answers "who are you?" |
| UUID generated by Auth0 at registration | No round-trip sync needed; service never generates player IDs |
| Lazy provisioning on first request | Player DB record is auto-created on the first authenticated API hit |
| `players.id` = Auth0-generated UUID | Single identifier across both systems; no separate `auth0_user_id` column |

## Actions

Actions run in the Auth0 pipeline on every login/registration event.
Navigate to **Actions → Library → Build Custom Action** for each one below.

### 1. Generate Player ID (Post-Registration trigger)

**Trigger:** Post User Registration
**Name:** `Generate Player ID`
**Runtime:** Node 18

Fires once when a user account is created. Generates the UUID that permanently
identifies this user's player record across the service. Stored in `app_metadata`
(server-editable, not user-editable — tamper-proof).

```js
const { randomUUID } = require('crypto');

exports.onExecutePostUserRegistration = async (event, api) => {
  // Only generate if not already set (idempotency guard for re-runs)
  if (!event.user.app_metadata?.player_id) {
    await api.user.setAppMetadata('player_id', randomUUID());
  }
};
```

**Deploy and add to the Post User Registration flow.**

---

### 2. Stamp Player ID Claim (Post-Login trigger)

**Trigger:** Post Login
**Name:** `Stamp Player ID Claim`
**Runtime:** Node 18

Fires on every login. Reads `player_id` from `app_metadata` and adds it as a
namespaced custom claim on the access token. The claim namespace must be a URL
— Auth0 rejects non-namespaced custom claims on access tokens.

```js
exports.onExecutePostLogin = async (event, api) => {
  const playerId = event.user.app_metadata?.player_id;

  if (!playerId) {
    // Safety net: player_id missing means the Post-Registration Action did not
    // run (e.g. user pre-dated the Action, or was created via Management API).
    // Log and skip — the service will return 403 until backfill is done.
    console.warn(`player_id missing for user ${event.user.user_id}`);
    return;
  }

  api.accessToken.setCustomClaim(
    'https://bowling-tournament-api.com/player_id',
    playerId
  );
};
```

**Deploy and add to the Login flow, after any existing Actions.**

## Custom claims reference

| Claim key | Source | Value |
|---|---|---|
| `https://bowling-tournament-api.com/player_id` | `app_metadata.player_id` | UUID — maps to `players.id` in the service DB |

Standard OIDC claims also available in the access token (configure under API settings):

| Claim key | Value |
|---|---|
| `sub` | Auth0 user ID (`auth0\|...`) — not used as a DB key |
| `name` | Display name — used for lazy player provisioning |
| `email` | Email — used for lazy player provisioning |
| `email_verified` | Boolean |

To include `name` and `email` in the access token (not just the ID token),
add them under **APIs → Bowling Tournament API → Settings → Access Token Claims**.
Alternatively, read them from the ID token if your client has access to it.

## Backfill: users created before the Action existed

Any user created before the `Generate Player ID` Action was deployed will have
no `player_id` in `app_metadata`. Their JWTs will arrive at the service without
the claim, and `requirePlayer` middleware will return 403.

Backfill script (run once via the Auth0 Management API or a local script using
the `auth0` Node SDK against your tenant):

```js
// Pseudocode — implement using auth0 Management API client
const users = await managementClient.getUsers({ per_page: 100, page: 0 });
for (const user of users) {
  if (!user.app_metadata?.player_id) {
    await managementClient.updateAppMetadata(
      { id: user.user_id },
      { player_id: randomUUID() }
    );
  }
}
```

Run this before enabling `jwtCheck` in the service.
